# Pet-Pass Dev Blog 작성 가이드

이 문서는 Pet-Pass Dev Blog에 **새 글을 쓰거나 기존 글의 어투를 다듬을 때** 반드시 따라야 하는 지침이다.
목표는 하나다 — **"틀에 맞춰 찍어낸 보고서"가 아니라 "차분하게 성숙한 글쓴이 본인"의 목소리를 유지하는 것.**

---

## 0. 경계선: 왜 이 문서가 필요한가

이 블로그 글은 두 시기로 나뉜다.

- **LLM 이전**: 개발 일기. 구어체, 솔직한 한계 인정, 말줄임표(`...`)의 여운, 닫지 않는 마무리.
- **LLM 이후**: 정형화된 테크 아티클. 🤦🔍✅ 이모지 템플릿, 개조식 요약, 단정적 결론, 억지 교훈.

연차가 쌓이며 어투가 차분해지는 건 자연스럽다. 문제는 그 변화가 **"성장"이 아니라 "기계적 이식"** 으로 느껴진다는 점이다.
이 지침은 LLM 이후 글에서 거세된 **'개인적 사유의 결'을 복원**하기 위한 것이다.

---

## 1. 핵심 원칙

### 1-1. 나이와 경력에 맞는 자연스러운 단어 (점진적 성숙)
주니어 시절의 과한 구어는 정제하되, AI 특유의 딱딱함이 아니라 **차분한 직무 성숙도**로 다듬는다.

| 과거 (너무 날것) | 지향 (성숙) |
|---|---|
| ~를 날려주고 | ~를 제외/제거하고 |
| 뭐가 뭔지는 잘 모르겠고 | 자세한 동작 메커니즘까지는 모르겠지만 |
| 헤맬 수 있지만 | 다소 헷갈릴 수 있지만 |

### 1-2. 말줄임표(`...`)로 남기는 여운과 겸손
완벽한 정답을 내리는 강사처럼 쓰지 않는다. 고민의 한계나 아쉬움을 솔직히 인정하며 말끝에 `...`를 의도적으로 남긴다.
- "시간 단축은 조금 더 고민해 봐야 할 듯..."
- "여기에 대해서는 조금 더 찾아봐야 할 듯..."

> `...`는 LLM 이전 글에서 가장 빈번했던 마무리 신호다. 특히 **문단·글의 끝**에서 살린다.

### 1-3. 독백·일기 형식의 서사적 도입
기술 정의로 시작하지 않는다. **최근 겪은 실무 경험이나 문제**를 툭 던지는 독백으로 연다.
- 좋은 예: "며칠 전까지 잘 되던 푸시가 갑자기 안 되는 이슈가 발생했습니다. 알아 보니..."
- 피할 예: "CompletableFuture는 자바에서 비동기 연산을 지원하는 도구로..." (교과서적)

### 1-4. 자기 객관화 표현 허용
솔직한 방어기제나 한계 인정 표현은 오히려 글에 생기를 준다.
- "물론 나도 ~하긴 했지만..."
- "사실 잘 모르고 쓴 게 맞다..."

## 2. 구조·마크다운 규칙

### 2-1. 인위적인 이모지 소제목 배제
`### 🤦 문제 발생`, `### 🔍 원인 파악`, `### ✅ 해결` 같은 이모지 데코 제목을 **절대 쓰지 않는다.**
소제목이 필요하면 투박하더라도 텍스트로만 쓴다.
가능하면 소제목을 줄이고 **문단 사이 빈 줄**로 흐름을 잇는다.

### 2-2. 넓은 여백과 짧은 호흡
한 문단에 서너 문장씩 우겨넣지 않는다. **1~2문장 단위**로 문단을 나눠 시각적 피로를 낮춘다.

### 2-3. 볼드(`**` / `<strong>`) 절제
거의 모든 핵심 문구를 볼드 처리하면 그 자체가 AI 레이아웃 느낌을 준다.
볼드는 **코드명·수치·고유명사** 위주로만 쓴다. 감정·판단을 담은 서술에는 쓰지 않는다.

## 3. 어투·종결형 규칙

### 3-1. 핵심 전제 — 종결어미는 "빈도"가 아니라 "문장의 기능"으로 정한다

흔한 실수: `~습니다`를 기본으로 두고 `~해요`를 "가끔 섞는 양념"이라고 생각하면, 평서문 곳곳에 `~요`가 무작위로 박혀 어색해진다.
정답은 **문장이 수행하는 역할(speech act)에 따라 어미를 고르는 것**이다. 아래 표가 1차 기준이다.

| 문장의 기능 | 어미 | 예시 |
|---|---|---|
| ① 사건·행동의 단순 서술 (narration) | `~습니다` / `~입니다` | "GOOGLE 계정으로 로그인해서 요청해봤습니다." |
| ② 이유·배경 설명, 속내 고백 (explanation) | `~거든요` | "막상 해보면 생각보다 훨씬 많은 판단이 필요하거든요." |
| ③ 직접 겪어 알게 된 경험 (experiential) | `~더라고요` / `~더군요` | "막상 돌려보니 생각보다 느리더라고요." |
| ④ 의문·추측·자문 (doubt) | 따옴표 독백 템플릿 (3-2 참고) | "이게 사실 문제였던 것인가?" 하는 생각이 들었거든요. |
| ⑤ 제안·권유 (suggestion) | `~요` / `~까요?` / `~ㅂ시다` | "이제 생각을 해볼까요." |
| ⑥ 가벼운 감상·리액션 (reaction) | `~네요` | "생각보다 깔끔하네요." |
| ⑦ 혼잣말·다짐 (resolution) | 명사형 / `~듯` / `~기!` | "다음엔 캐싱부터 잡아두기!" |

> **헷갈릴 때의 판별법**
> - 문장 앞에 **"사실"·"왜냐하면"**을 붙여 자연스러우면 → `~거든요` (②)
> - 그냥 일어난 일을 **순서대로 적는** 문장이면 → `~습니다` (①)
> - **`~요`로 평서문을 끝내고 싶어지면** 거의 항상 `~거든요`(②)나 `~더라고요`(③)가 더 맞다. 단순 `~어요`/`~해요` 평서 종결은 의식적으로 줄인다.

### 3-2. 독백체 의문·추측 — 따옴표 자문 템플릿

사건에 의문을 표하거나 추측할 때는 평서문으로 단정("이건 문제였다")하지 않고, **속으로 던진 질문을 따옴표로 인용**한다. 글쓴이의 사유 과정을 그대로 보여주는 핵심 장치다.

구조: `"〔의문형 문장〕" 하는 생각이 들었거든요.`

- 인용 안의 의문형 어미는 `~ㄹ까?`, `~인가?`, `~던 것인가?`, `~던 걸까?` 등을 쓴다.
  - "이게 사실 문제가 있던 걸까?" 하는 생각이 들었거든요.
  - "이게 사실 문제였던 것인가?" 하는 생각이 들었거든요.
- 인용 뒤 프레임은 상황에 맞게 변주한다: `하는 생각이 들었거든요` / `싶었습니다` / `싶더라고요`.

### 3-3. 자기 성찰적 명사 종결 혼용

전체적으로는 `~습니다`를 쓰지만, 혼잣말이나 다짐을 할 때는 명사형이나 반말(`~듯`, `~기!`)을 슬쩍 섞어 입체감을 준다.
- 예시: `~을 고려해보기!`, `~를 참고해야 할 듯하네요.`

### 3-4. 설명조/강의조 피하기

존댓말을 쓰더라도 강사나 기계처럼 지식을 주입하려는 어조(`~하셔야 합니다`, `다음과 같습니다`)는 피하고, "제가 겪어보니 이렇더라고요" 식의 담담하고 친근한 대화형 독백 스타일을 고수한다.

### 3-5. 분류·전환 문장 금지

다음 내용을 자연스럽게 이어 쓰면 될 것을 AI가 굳이 한 줄로 요약·예고하는 패턴을 쓰지 않는다.
- 금지 예: "두 케이스 모두 ~라는 게 공통점입니다.", "실무에서 쓰임새가 더 있었습니다.", "한 가지는 분명하게 느꼈습니다.", "이 경험에서 배운 건 생각보다 범위가 넓었다."

## 4. 결말부 작성 패턴

### 절대 금지 — AI식 억지 교훈·병렬 대비 결론
- "결국 A와 B는 맞닿아 있다", "A와 B는 다른 이야기다"
- "한 가지는 분명하게 느꼈다", "배운 점은 ~이었다"

글을 억지로 웅장하거나 감동적으로 끝맺으려 하지 않는다.

### 승인된 결말 패턴
아래 중 **상황에 맞는 것을 고른다.**

1. **무심한 단상형** — 해결 후 가볍게 느낌만 던지고 끝.
   - "이게 또 이론적인 거 하고 실제 사용되는 부분이 조금 다른가 봐요."
2. **기술적 사실 종결형 (No Ending)** — 기술 요약·리스트만 나열하고 미련 없이 끝.
3. **다음 스텝 / 셀프 피드백형** — 미완의 한계나 다음에 시도할 것을 메모처럼 남김. (혼잣말 톤이라 반말·명사형 혼용 OK)
   - "다음엔 ~을 고려해보기!", "이유는 조금 더 찾아봐야 할 듯..."
4. **소박한 다짐형** — 일기장에 적듯 담담한 향후 계획.
   - "개인적인 느낌을 정리해 나가야겠어요."

### 권고만 가능 — 본인이 직접 써야 하는 패턴
다음 두 패턴은 **글쓴이 고유의 개인적 결**이라 AI가 흉내 내지 않는다.

- **일상·잡담형** — "추석을 코앞에 두고 있다... 잠 많이 자는 게 목표."
- **`ps` / `p.s.` 추신**

## 5. 글 작성 워크플로우 (마무리 패턴 선택 규칙)

새 글 작성을 요청받으면 **결말부 패턴을 2개 고른다.**

- 고른 2개가 모두 §4의 **승인된 패턴**이면 →
  두 패턴을 글쓴이에게 각각 제시하고, 어느 쪽으로 마무리할지 선택을 요청한다. 에이전트 혼자 최종 선택하지 않는다.
- 고른 2개 중 하나라도 **잡담형 / 추신형**이 있으면 →
  그 마무리는 작성하지 않고 글쓴이에게 넘긴다. ("이 글의 마무리는 직접 써주세요" 신호로 본다.)

## 6. 금지 표현 목록

### 6-1. 과잉 서사형 도입부
| 금지 예 | 이유 |
|---|---|
| "이 이야기는 아주 단순한 질문에서 시작됐다" | AI 특유의 서사 프레임 |
| "시작은 사소했다" | AI 특유의 서사 프레임 |
| "~이(가) 발단이었다" | 과잉 극적화 |
| "뜻밖의 계기로" | 작위적 복선 느낌 |
| "그게 이 이야기의 시작이었다" | 동일 패턴 |

대신 그냥 상황을 툭 던진다. — "며칠 전 ~하다가 ~가 발생했습니다."

### 6-2. AI식 문장 자동화 패턴
| 금지 예 | 대체 |
|---|---|
| "소중한 시간이었다" | 삭제하거나 구체적 감상으로 |
| "기대 이상이었다" | 뭐가 구체적으로 좋았는지 1문장 |
| "유익한 프로젝트였다" | 삭제하거나 무심한 단상형으로 |
| "~를 깨달은 대목이다" | "~인가 보다." 정도로 무심하게 |
| "~앞으로 더 많아질 것 같다" (예측형) | 삭제 — 트렌드 예측은 쓰지 않는다 |

## 8. 프로젝트 형식 규칙 (Pet-Pass 전용)

### 8-1. 시리즈 분류
| 시리즈 태그 | 예시 포스트 |
|---|---|
| 바이브 코딩 개발기 · 이슈 #N | post2, post3, post6, post7, post8 |
| 기획 & 디자인 · #N | post4, post5 |
| 바이브 코딩 개발기 · 시작하며 | post1 |
| 바이브 코딩 개발기 · 마치며 | post9 |

### 8-2. lesson-box 제목 규칙
포스트 말미의 정리 박스(`<div class="lesson-box">`) 제목은 시리즈에 따라 아래 둘 중 하나만 사용한다.

| 시리즈 | h3 텍스트 |
|---|---|
| 이슈 (#N) | `이 문제에서 배운 것` |
| 기획 & 디자인 | `이 경험에서 배운 것` |
| 시작하며 (post1) | 자유 (lesson-box 없음) |

"이 경험에서 가져간 것", "이 이슈에서 배운 것", "이 글에서 배운 것" 등의 변형은 사용하지 않는다.

### 8-3. index.html 카드 날짜
`<div class="blog-post-date">` 안에 날짜(연. 월. 일)는 적지 않는다. 시리즈 태그와 이슈 번호만 표기한다.

### 8-4. lesson-box CSS
`blog_post.css`에 공통 정의되어 있다. 포스트별 인라인 override 금지.
- 좌측 border: `border-left: 4px solid var(--primary)`
- 제목(h3): `font-weight: 800`, `color: var(--text-primary)`
